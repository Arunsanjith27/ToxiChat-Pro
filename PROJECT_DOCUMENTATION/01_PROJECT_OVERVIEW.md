# ToxiChat Pro - Project Overview

## Project Purpose
ToxiChat Pro is a real-time, AI-moderated chat application designed to enforce community guidelines, ensure compliance, and proactively manage conversational risk. Unlike standard messaging platforms, ToxiChat actively analyzes text, image, and audio messages in real time using a suite of multi-modal machine learning pipelines to detect toxicity, profanity, Personally Identifiable Information (PII), and Not-Safe-For-Work (NSFW) content.

## Real-World Problem Solved
Digital platforms face immense challenges regarding cyberbullying, data exfiltration, and non-compliance with corporate communication standards. Manual moderation is unscalable, slow, and reactive. ToxiChat solves this by introducing a highly proactive layer of AI moderation that intervenes *before* conversations escalate, providing real-time text rewriting suggestions, muting abusive users automatically, and generating structured incident evidence packages for human review.

## Target Users
1. **Administrators & Compliance Officers**: Require audit trails, high-level dashboards, incident timelines, and automated compliance reports.
2. **Moderators**: Benefit from the "Moderator Copilot," which summarizes long threads, provides contextual alerts, and identifies high-risk users.
3. **General Users**: End-users participating in DMs or Group Chats who receive a safer, cleaner communication environment with real-time feedback on their messaging tone.

## Business Value
- **Risk Mitigation**: Reduces legal liability by preventing PII leaks and maintaining a hostility-free workplace.
- **Cost Reduction**: Automates tier-1 moderation, drastically reducing the human capital required to police large-scale communities.
- **Actionable Intelligence**: Turns unstructured chat data into structured escalation predictions and conversational health metrics.

## Architecture Philosophy
ToxiChat follows a **modular monolith** design pattern, built to transition seamlessly into microservices if scaling is required. 
- **Separation of Concerns**: The AI inference engine (Manager) is decoupled from the WebSocket routing and standard CRUD APIs.
- **Graceful Degradation**: The AI pipelines employ fallback mechanisms (e.g., if Redis or the ML models fail, messages are still routed but flagged for asynchronous review, or the system falls back to regex-based heuristics).
- **Event-Driven**: Message ingestion immediately fans out to the database layer, the connected clients via WebSockets, and the async ML pipelines.

## Technology Stack

### Backend
- **FastAPI**: Chosen for its high performance (Starlette), native async/await support, and built-in Pydantic validation.
- **MongoDB (Motor)**: Document-oriented NoSQL storage allows for flexible message schemas (e.g., attaching complex AI analysis trees directly to the message object).
- **Redis**: Used for high-speed pub/sub messaging across distributed WebSocket workers, caching, and rate-limiting.
- **WebSockets**: Facilitates real-time, bi-directional communication for messages, typing indicators, and presence tracking.

### Frontend
- **React (Create React App / Webpack)**: Provides a robust component-based UI architecture.
- **TailwindCSS**: Utility-first CSS framework chosen for rapid, responsive UI development without context switching.
- **Lucide React**: Vector icons used consistently throughout the dashboard and chat interfaces.

### Machine Learning & AI
- **Text (Toxicity/Emotion)**: RoBERTa-based models (via HuggingFace) or localized heuristics depending on environment.
- **Audio (Transcription/Toxicity)**: `faster-whisper` for highly efficient CPU/GPU audio transcription.
- **Vision (NSFW/Violence)**: `nudenet` leveraging ONNX runtime for explicit content detection.
- **OCR (Text Extraction)**: `easyocr` for extracting text embedded within images before passing it to the text moderation pipeline.
- **Semantic Search**: Vector embeddings for context-aware message retrieval.

## Alternative Technologies
- **PostgreSQL vs MongoDB**: Postgres was a strong alternative, but MongoDB's schema-less nature better suits the evolving, deeply nested JSON outputs generated by the AI pipelines (e.g., `explanation`, `pii_entities`, `vision_scores`).
- **Node.js vs FastAPI**: While Node.js is typical for WebSockets, FastAPI was selected because Python is the native ecosystem for Machine Learning integration (PyTorch, ONNX, HuggingFace), preventing the need for an external microservice right out of the gate.

## Limitations & Future Scalability
- **Limitation**: Currently, ML models run in the same process space as the FastAPI web server. Heavy loads on Whisper or OCR will block the event loop or consume excessive CPU, degrading WebSocket performance.
- **Scalability Path**: The `ai.manager` layer should be extracted into a dedicated GPU-backed Celery/RabbitMQ worker queue.
- **Limitation**: In-memory fallback database exists for development but poses data-loss risks if MongoDB is misconfigured in production.
- **Scalability Path**: Horizontal scaling of the FastAPI instances behind a load balancer, relying strictly on Redis Pub/Sub for cross-instance WebSocket broadcasting.
